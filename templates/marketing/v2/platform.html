§ 01 — the world model layer

A graph that tracks state, not text.

What an LLM does well — predicting plausible next tokens — is not what a script supervisor needs. A script supervisor needs to know what is true in the show right now, and what was true three episodes ago, and what each character knew when they made a decision.

Where most tools collapse a show into a single timeline of dialogue, Fabula gives every character, location, organisation, and object a queryable journey. Every event a character participates in carries that character's beliefs, goals, and emotional state at that moment — a stack of psychological states down each character's spine, all anchored in the Episode → Act → Scene → Beat structure of the show.

“What did Bartlet believe when he made this decision?”
“What had he done by this point in the season?”
“What changed in him between this season and the next?”

Different operations. All answerable on the same per-entity arc.

§ 01a — universal by construction

Works on a show the model has never heard of.

The schema was designed before any show was loaded, and it's the same schema for every show that's gone through since — no per-series fine-tune, no genre-specific config, no “this works because the model has already read the Wikipedia article.” Every model call gets only the narrative evidence presented in context: excerpts of script, prior graph state, the shape of the question. The judgement is forced onto what's on the page.

That matters in two directions. For an unaired pilot, a niche drama in a language with no English-language coverage, or any show the model couldn't possibly know — the system performs the same as it does on TNG. For high-profile IP that is in training data — the model can't quietly lean on what it remembers from the internet. Either a claim is justified by the lines of script Fabula handed it, or it isn't.

The entity adjudicator works the same way. When it has to decide whether two similarly-named characters are the same person, it reads the lines around each, looks at graph state for both, and writes a paragraph of reasoning. If the evidence is ambiguous, it keeps them separate and flags it for review. It does not infer from training-data familiarity. (We've written about the design of this loop at length: teaching a knowledge graph to learn from “no.”)

Atomic checkpointing after every episode. Three-level error recovery. Bounded async concurrency. Circuit breakers with model fallback. § 03 — model selection

The best model for each job.

Different parts of the pipeline have different shapes of failure. Scene decomposition is mostly structural and runs cheaply on Flash. Entity adjudication needs better judgement and runs on Sonnet, with a fallback to GPT-4 when Sonnet's confidence drops below threshold. Theme analysis runs on whichever model produces the most useful output for that show — we benchmark per-genre.

Because the prompt for each step gets a tight, engineered context rather than the whole episode, the cheaper models do well at their job. The expensive ones only run where they earn it. (More on what happens after the model returns, in post-processing as ontology maintenance.)

The same property that lets cheap cloud models pass — small focused tasks, small focused payloads — lets local open-weight models stand in. For studios whose unfilmed scripts can't leave the building, any step in the pipeline can be pointed at a Llama, Mistral, or Qwen model running on-prem. The schema doesn't change; the audit trail doesn't change; the only thing that changes is where the inference happens.

And it's the same property that lets the platform compound with the field. Models ship every few weeks; one of them is going to be sharper at scene segmentation, another better at disambiguating which John walked into the room, another better at writing a per-episode synopsis a human will actually read. Each of those upgrades is a config line, not a rewrite — schema unchanged, audit trail unchanged, the extractions and linkages and prose all tighter than the run before. Designing for the next model is how a system avoids dating itself to the one it shipped on.

§ 04 — open by default

The catalogue graphs are private. The story graph shouldn't be.

Streaming platforms keep their entertainment knowledge graphs internal because their value is in recommendation — ranking titles you might watch next. That's a reasonable choice for that data.

The graph inside the show is different. It's reference material. Production teams need to be able to take it with them between studios. AI builders need to be able to retrieve from it with their own infrastructure. Game studios need to be able to bulk-export it into their own pipelines. So Fabula's data leaves the platform, in formats people already use: Parquet for analytics, Cypher for Neo4j, plain JSON for everything else.

Fifty-six datasets are already published openly under brandburner on Hugging Face — including the Doctor Who megagraph (703 episodes, 337,795 relationships) and per-series megagraphs for TNG, The West Wing, Wolf Hall, and Happy Valley. The example below is the export command that produced one of them. 

$ fabula export the_west_wing --season 4 --format parquet
